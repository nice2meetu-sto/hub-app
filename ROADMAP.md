## 1단계: 플레이스토어 출시 (TWA)

**개발 형식은 거의 안 바꿔도 됨.** 이미 PWA(manifest + 서비스워커 + HTTPS)라서
구글 공식 지원 방식인 **TWA(Trusted Web Activity)** 로 포장만 하면 된다.

- 도구: `Bubblewrap`(CLI) 또는 `PWABuilder`(웹) — 기존 웹앱을 여는 안드로이드 껍데기 앱을 자동 생성
- 웹앱은 GitHub Pages에 그대로 → **출시 후에도 git push만 하면 즉시 업데이트** (스토어 재심사 불필요)
- 추가로 필요한 것:
  - `assetlinks.json` (도메인 소유 증명 파일 1개)
  - 앱 서명 키
  - 스토어 등록 정보: 스크린샷, 설명, **개인정보처리방침 URL**
- 비용/절차:
  - 구글 플레이 개발자 계정 **$25 (1회)**
  - ⚠️ 개인 계정 신규 등록은 **비공개 테스트(테스터 12명 × 14일 연속)** 통과 후 정식 출시 가능 → 동호회 멤버로 채우면 됨
- Capacitor/React Native 재개발은 불필요. 푸시 알림 등 네이티브 기능이 필요해지면 그때 Capacitor 고려.

## 2단계: 멀티허브 + 하이브리드 인증 개편

### 2-α. 개발 방식: 별도 저장소 샌드박스 (결정)

기존 사용자 무중단을 위해 **개발 전용 저장소 + 개발 전용 Supabase 프로젝트**에서 작업:

1. 새 저장소(예: `boardgamehub-dev`)에 현재 파일 전부 복사 → 자체 Pages 주소로 자유롭게 테스트
2. **개발용 Supabase 프로젝트 별도 생성** (무료 플랜 2개 가능) — 스키마 대공사가
   운영 DB/실사용자 데이터를 건드리지 않도록. 개발 저장소의 URL/anon key만 교체
3. 완성 후 복귀: dev 코드를 **본 저장소로 다시 합치고** + 운영 DB에 마이그레이션 SQL
   (기존 데이터 → "1번 허브") → **앱 주소 불변** = 설치된 PWA·TWA(URL에 묶임) 무중단 업데이트
4. 새 주소로 갈아타기는 금지 (설치된 앱 전부 끊김)

### 2-0. 개편 첫 단추: 파일 분리 (준비 작업)

- 개편 시작 전에 `index.html` → `index.html`(마크업) + `styles.css` + `app.js` 3파일로 분리
- 기능 변화 0, 빌드 도구 불필요 (`<link>` + `<script src>`만) — GitHub Pages 그대로
- 서비스워커 SHELL 목록에 두 파일 추가 + 캐시 버전 올리기
- 분리 직후 정상 동작 확인하고 나서 본 공사 시작 (문제 원인 분리 용이)
- JS를 기능별 모듈로 더 쪼개는 건 이번엔 안 함 (3파일이면 충분)

### 2-1. 인증 모델: 하이브리드 (결정 사항)

전원 이메일 로그인 대신 **관리자만 Auth, 멤버는 지금처럼 닉네임+PIN 유지**.

| 단계 | 로그인 | 할 수 있는 것 |
|---|---|---|
| 게스트 | 없음 | 플레이 참가자로 이름만 기록 (현행 유지) |
| 멤버 | 닉네임+PIN | 허브 안 모든 활동 (현행 유지) |
| 이메일 연동 멤버 (선택) | 이메일(Supabase Auth) | + 여러 허브 통합 기록, 탈퇴 후 기록 보존, 관리자 후보 자격 |
| 허브 관리자 | 이메일(Supabase Auth) 필수 | + 멤버/게임/기록 관리, 삭제 등 파괴적 작업 |

선택 이유: 동호회 앱은 가입 마찰이 생사를 가름. 멤버 데이터는 민감정보가 아니고,
진짜 위험한 권한(삭제·멤버 관리)만 Auth로 잠그면 위험 대부분이 해소됨.

**하이브리드 필수 보안 3종 세트** (같이 구현해야 함):
1. **허브 가입 초대코드** — 외부인 사칭 차단 1차 방어선 (제일 중요)
2. **PIN 시도 횟수 제한** — 4자리는 1만 조합뿐. RPC에서 "5회 실패 시 10분 잠금" 등 (시도 기록 테이블 + RPC 수정)
3. **파괴적 작업은 전부 관리자(Auth) 전용** — 현재 admin 경계 유지

### 2-2. 스키마 설계

```
hubs:    hub_id, 이름, 초대코드, owner_uid(Auth ID)
        (owner 1명 + 공동 admin 여러 명 구조 권장 → hub_admins 테이블)

players: 기존 + hub_id, auth_uid(nullable), status('active'|'left')
playlogs / ratings / categories: 기존 + hub_id
games: 공용 도감 (hub_id 없음) — 아래 2-7 참고
hub_games(허브 선반): hub_id, game_id, 허브별 카테고리

닉네임 유일성: 전체 → 허브 단위로 변경
  create unique index players_hub_name on players (hub_id, lower(name));

RLS/RPC: 모든 조회·쓰기를 hub_id 범위로 제한
기존 데이터: "1번 허브"로 그대로 이전 (기록 손실 없음)
```

- 신규 스키마 짤 때 `players.auth_uid`(nullable)와 `players.status`는 **처음부터 포함**할 것
  → 이메일 연동/탈퇴 기능을 나중에 붙여도 스키마 재공사 불필요

### 2-3. 허브 관리자 교체

- 관리자 권한은 사람이 아니라 **허브의 속성**(`hubs.owner_uid`)
- 절차: 새 관리자가 이메일 Auth 계정 생성 → 현 관리자가 관리자 페이지에서 [관리자 넘기기] → `owner_uid` 변경 (현 owner 로그인 상태에서만 실행되는 RPC)
- 이전 관리자는 일반 멤버로 강등 또는 공동 admin으로 잔류
- **플레이 기록은 player_id에 묶여 있어 관리자 교체와 무관** (데이터 이동 없음)
- 관리자 잠수 시 비상구: 서비스 운영자가 Supabase 대시보드에서 `owner_uid` 수동 변경 (앱 기능으로 안 만듦)

### 2-4. 닉네임 & 로그인 흐름 변화

- 닉네임은 **같은 허브 안에서만 유일** (다른 허브에 동명 가능)
- 파급효과: 로그인 시 허브 컨텍스트 필요
  1. 최초 1회: 초대코드 → 허브 확정 → 닉네임+PIN
  2. 이후: 기기가 허브 기억(localStorage) → 지금과 동일하게 닉네임+PIN만
  3. 복수 허브 소속자: 로그인 화면에 허브 선택 드롭다운

### 2-5. 허브 탈퇴 / 복수 허브 / 이메일 연동 통합 기록

**열쇠고리 모델**: 이메일 계정 = 여러 허브의 나를 하나로 묶는 열쇠고리

```
이메일 계정 (선택)
 ├── A허브의 '재석'  (players.auth_uid로 연결)
 └── B허브의 '유산슬' (허브마다 닉네임 달라도 무관)
```

- **연결 절차**: 이메일 가입 → 해당 허브에 닉네임+PIN으로 로그인한 상태에서 [이 기록을 내 계정에 연결] (PIN을 아는 것 = 본인 증명)
- **복수 허브 동시 가입**: player가 허브마다 따로 있어 충돌 없음
- **허브 탈퇴**: player 삭제 ❌ → `status='left'` 소프트 탈퇴 ⭕
  - 허브 쪽: 과거 기록·순위·통계 보존 (삭제하면 남은 멤버 전적 왜곡)
  - 멤버 목록·참가자 자동완성에서만 제외
  - 본인 쪽: 이메일 연동 시 탈퇴한 허브 기록도 통합 기록에 계속 표시
- **통합 조회 RLS**: `auth_uid = auth.uid()`인 기록만 크로스 허브 조회 허용

### 2-6. UI 설계

**원칙: 허브는 "공간", 이메일 계정은 "사람". 플레이·게임 탭 = 공간, MY 탭 = 사람.**

- **MY 탭을 계정 중심으로 승격**:
  ```
  MY 탭
   ├─ 상단: 허브 필터 칩 [ 전체(통합) | 허브A | 허브B ]
   ├─ 전체 기록 / 플레이 기록 / 게임 기록 (기존 서브탭 그대로)
   └─ 필터에 따라 통합 ↔ 허브별 전환
  ```
  - 이메일 미연동 멤버: 필터 자체가 안 보임 → **지금 화면과 100% 동일** (기존 UX 불변)
  - 기존 통계/그래프/게임별 리스트 UI를 데이터 범위만 바꿔 재사용 → 개발량 절감
- **플레이/게임 탭은 항상 현재 허브 기준** + 앱바 제목 = **"{허브이름} Hub"** (예: 슈필 Hub)
  - 제목 탭 → 허브 전환 메뉴 (전환 버튼 겸용, 현재 허브 오인 기록 방지)
  - 홈 화면 앱 아이콘 이름은 "보드게임 Hub" 고정 (manifest) — 앱 안 제목만 동적
- **첫 진입 화면 분기**:
  ```
  [ 허브로 시작하기 ]  → 초대코드 → 닉네임+PIN 가입/로그인
      · 허브 만들기    → 이메일 가입(관리자) → 허브 개설
  [ 개인으로 시작하기 ] → 이메일 가입/로그인 → MY(통합)
  ※ 기기에 기억 → 다음부턴 바로 자기 화면
  ```
- **허브 없는 개인 회원**: "개인 기록장"을 **첫 배포에 포함** (결정)
  - 개인 가입 시 내부적으로 **1인 전용 허브 자동 생성** — 기존 허브 로직 100% 재사용
  - 허브 없이도 혼자 플레이 기록 가능, 나중에 동호회 가입하면 통합 기록에 자연 합산
- **기록 작성 원칙: "보는 건 통합, 쓰는 건 허브 단위"**
  - 플레이 기록은 항상 **현재 허브**에 귀속 (+버튼 = 현재 허브에 기록, 매번 안 물어봄)
  - 참가자·게임 선반·자동완성이 전부 허브에 묶여 있어 기록은 태생적으로 한 허브 소속
  - 통합(전체) 화면에서 + 누르면 "어느 허브에 기록할까요?" 선택 팝업 → 전환 후 기록
- **허브 이름**: 자유 작명, 관리자가 언제든 변경, **중복 허용**
  (가입은 초대코드로 하므로 이름은 식별자가 아니라 간판. 공개 검색 기능이 생기면 그때 태그 구분 도입)

### 2-7. 게임 등록: 공용 도감 + 허브 선반 (결정 사항)

허브별 개별 등록(안 2) 대신 **게임은 공용, 노출은 허브 선반 기준**(안 1 보완형)으로 간다.

```
games (공용 도감)     : 이름·인원수·시간·이미지·요약 — 전 허브 공유
hub_games (허브 선반)  : hub_id, game_id, 허브별 카테고리
plays / ratings / 후기 : 지금처럼 전부 허브 소속 ('우리Hub평점' 유지)
```

- **게임탭에 보이는 것 = 우리 허브 선반에 있는 것.** 선반 등재 경로 2가지:
  1. 허브에서 게임 등록 시 (신규 등록이든 도감에서 끌어오기든) 즉시 선반 추가
     → "미리 등록해둔 게임이 안 보이는" 문제 방지
  2. 플레이 기록 작성 시 자동으로 선반에 추가 (플레이 기반 노출)
- **게임 추가 UX**: 이름 입력 → 공용 도감 자동완성 → 있으면 "우리 허브에 추가" 원터치,
  없으면 신규 등록. 기존 중복 검사(띄어쓰기·유사이름)가 전역 검사로 승격됨.
- **카테고리는 전역 공통 관리로 결정 변경(2026-07)**: 분류 목록은 서비스 공통 7종
  (전략/마피아/트릭테이킹/1대1 게임/카드게임/경매게임/협력게임),
  게임의 분류는 도감(games.category)에 저장 → 끌어오기 시 분류도 연동
- **공용 도감을 택한 결정적 이유**: 개인 통합 기록의 게임 단위 합산
  ("내 카탄 통산 전적")은 허브들이 같은 game_id를 공유해야 정확함
- **공용 정보(요약·이미지) 수정권** (초기 정책):
  - 사용 허브가 1개 → 그 허브 관리자가 자유 수정
  - 여러 허브 사용 → 운영자만 수정 (추후 '수정 제안' 기능으로 발전 가능)
- 허브에서 게임 "삭제" = 선반에서 내리기 (공용 도감에서는 삭제 안 됨)

## 3단계: Supabase Pro 전환

- **초기엔 무료 유지 가능** — 500MB DB·5GB 트래픽이면 동호회 수십 개 규모까지 여유. keep-alive 워크플로 가동 중(6일 간격, `.github/workflows/supabase-keepalive.yml`)
- **Pro($25/월) 전환 시점**: 실제 외부 사용자가 붙기 시작할 때
- Pro의 실질 이득: ① 7일 미사용 일시정지 없음 ② **일일 자동 백업 7일치** (외부 사용자 데이터 받으면 필수) ③ DB 8GB·트래픽 250GB ④ 지원
- 아키텍처(anon key + RLS + SECURITY DEFINER RPC)는 플랜과 무관하게 그대로 유지
